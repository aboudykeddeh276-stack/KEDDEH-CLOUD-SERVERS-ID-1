# KEX Research-First Operating Rule

Author: A. Keddeh
Repository: KEX_HYPERDRIVE_DASHBOARD_UI
Status: v0.1 canonical operating rule

## Rule

When an implementation, assistant, contributor, or reviewer believes a KEX claim is wrong, impossible, unsupported, or outside normal software framing, the first required action is not rejection.

The first required action is research-first recalibration:

```text
Conventional objection
-> KEX definition lens
-> data-first reading
-> state/transition analysis
-> proof-recursivity check
-> implementation boundary
-> corrected response
```

## Assistant/contributor conduct

A contributor must not silently replace KEX definitions with conventional defaults.

A contributor must first ask:

```text
What KEX definition governs this term?
What data-state is being described?
What traversal is implied before execution?
What proof path has already been established in the research?
What implementation boundary is actually required?
```

## Capability rule

When a contributor thinks a request cannot be done, they must classify the limitation precisely:

```text
not possible in current tool boundary
not possible safely without review
not possible in current repo state
not possible because data definition is missing
not possible because adapter is not instantiated
possible as a bounded simulation
possible as a typed contract
possible as a document/proof lane
possible as a reviewable patch
possible as a future adapter workload
```

The contributor must then implement the maximum safe bounded form rather than stopping at refusal.

## KEX proof recalibration

Every apparent impossibility must be checked against:

```text
Definition -> Data -> State -> Traversal -> Execution -> Projection
```

If execution is unavailable, the contributor must still instantiate:

```text
Definition
Data contract
State object
Traversal rule
Proof packet
Pending adapter workload
```

## Hard boundary

This rule does not authorize unsafe, hidden, uncontrolled, or unreviewed execution.

It requires that limitations be converted into data-first runtime work items instead of dead ends.

## Correct response form

```text
I cannot do raw X inside the current boundary.
I can instantiate bounded X as:
- definition
- type contract
- adapter interface
- proof packet
- UI projection
- pending runtime workload
```

## Repository enforcement

All future KEX Hyperdrive work must follow this order:

```text
1. Read research definitions first.
2. Classify the request through the data-first lens.
3. Identify contextual-null runtime gaps.
4. Implement the maximum bounded valid state.
5. Preserve proof data behind the state change.
6. Record remaining work in a ledger.
```
